# 11. MVP-0.5 — Dev doc vs Test doc 比对报告

> 比对 `11-mvp-0.5-spec.md` (dev doc, 894 行) vs `11-mvp-0.5-tests.md` (test doc, 1353 行)。
> Subagent 黑盒视角写 test doc 完成后的差异分析。
> 找 truth gap：dev doc 承诺但 test 没覆盖 / test 假设但 dev 没承诺 / 两边说不同话。

---

## 0. 比对方法

1. 提取 dev doc 全部 commit (B-X-Y 行为 + Failure modes + 错误码)
2. 检查 test doc Coverage Map 是否对每个 commit 都有 test ID
3. 反向检查 test doc 假设的行为是否在 dev doc / surface 显式承诺
4. 找 dev doc 提了但 surface 没暴露 → 即 surface gap
5. 找 test doc §H 自己 declare 的"难黑盒验"清单 → 那是 test writer 主动 flag 的盲区

## 2. 真实 Truth Gap

下表列出 dev doc vs test doc 的**实质性不一致**。每条标 severity (P1/P2/P3) + 修哪边。

### Gap 1 (P1) — Engine output JSON schema 未暴露到 surface

**Test doc §H 第 12 行原话**:
> public surface 没暴露 schema 形状；我只知道 "schema 错"，不知道 valid schema 长啥样

**Dev doc 实况**:
- §A.4 `LLMSubprocResult.parsed_output: unknown` ← unknown 类型
- §A.5 reviewer 调用 `runAlignmentProbe` 内部 parse 解析 schema，但 schema 形状只在 prompt template 里隐式存在
- §A.9 "Engine output 不符合 JSON schema → 5 retry" 用了 "schema" 这个词，但没指 schema 具体是什么

**实质**: dev doc 暗示了"schema" 存在但**没规定**。Public surface 也没暴露。Test 写作者只能反向断言"parsed_output is null"。

**修哪边**: **修 surface (P1)**。Schema 是契约不是实现细节——alignment probe / adversarial reviewer 各自的 JSON 输出 schema (字段名 / 类型 / 必选可选) 应该在 surface 显式列出。Dev doc 也应该把每层 reviewer 的预期 output schema 写到 §A.5 末尾。

**Action item**:
- 在 `11-mvp-0.5-public-surface.md` §A 加 `AlignmentVerdict` + `AdversarialVerdict` + `DoneCheckVerdict` 三个 schema 接口
- 在 `11-mvp-0.5-spec.md` §A.5 加对应的 schema 定义
- 然后让 test doc subagent 补 schema 形状的 positive test

---

### Gap 2 (P2) — Calibration check 调用 trigger 没暴露

**Dev doc §A.7 原话**:
> Calibration check 跑时机：`d2p doctor` 命令手动跑（用户可触发）; MVP-0.5 不强制每 session 跑; MVP-1+ 加 nightly cron

**Public surface 实况**: 只暴露 `runCalibrationCheck` 函数签名（B-A-8, B-A-9）。**没提**"d2p doctor 命令"作为 trigger。

**Test doc 实况**: T-A-8 / T-A-9 测函数本身。**没测** CLI `d2p doctor` 触发链。

**实质**: dev doc 把 CLI 触发当 commit，但 surface 没暴露 → test 没法测 → CLI doctor 命令上线后没 test 保护。

**修哪边**: **修 surface (P2)** —— 加 CLI 命令 `d2p doctor` 作为 public surface 一部分（外部用户可触发的行为）。然后 test doc 补 `T-A-doctor-cmd-triggers-calibration`。

---

### Gap 3 (P2) — 维护者 postmortem 30 天流程没法 enforce

**Dev doc §D.5 原话**:
> Maintainer 想紧急 bypass → 改 branch protection 手动 disable required check → 只允许 owner 一人有权限；改动记进 audit log；改完必须 30 天内 write postmortem

**Test doc §H 第 10 行 ack**:
> Dogfood-debt mode 维护者 postmortem 流程 (D.5 exit 条件) — 是流程契约不是代码契约 — 不写测

**实质**: "30 天内 postmortem" 是 dev doc 承诺，但**没有 code-level enforce 机制**——靠 maintainer 自觉。如果维护者不写 postmortem 没人发现。这是 v0.7 §9.6 governance 缺失的具体表现 (red line 6 "willpower-as-strategy")。

**修哪边**: **修 dev doc (P2)**。要么:
- (a) **承认这条是 willpower red line**，不能 enforce，跟 v0.7 §9.6 表述一致；或
- (b) **加 GitHub Action 30-day check**：watchdog 扫 `docs/dogfood/_postmortems/` 看有没有 30 天前 owner-bypass 没 follow-up postmortem，有 → 开 issue。这条 enforce 可以工程化，但 MVP-0.5 是否做要决定。

**Action item**: 在 `11-mvp-0.5-spec.md` §D.5 明确选 (a) 或 (b)。如果 (b)，写到 surface (B-D-13)，让 test doc 补对应 test。

---

### Gap 4 (P3) — Cap 触发 user-facing 报告内容质量未验

**Dev doc §E.6** (surface B-E-6) 提及 "graceful degradation 模式: partial-PASS 报告 + 列已完成 vs 未完成 + 让用户决定 ship-partial vs continue"。

**Test doc T-E-3-cost-cap / T-E-3-iter-cap / T-E-3-cap-raise** 测了:
- cap 触发条件（cost > $8 或 iter > 10）
- 触发后 daemon 行为（停 / 返回 partial-PASS）
- 用户提高 cap 后能继续

**没测**:
- "partial-PASS 报告"具体内容质量（用户能不能看懂）
- "ship-partial vs continue" UI 文案

**Test doc §H 第 11 行 ack**: "单测验 shape，不验 '用户能看懂' 质量 — UX review + 5 个真用户 dogfood 反馈"

**实质**: 这是 **UX/content quality** 类 contract，本来就不适合单元测——subagent 承认得对。

**修哪边**: **接受 §H 11 行 ack**，不做改。可选：加 UX review checklist 到 `docs/dogfood/_INVARIANT.md`。

---

### Gap 5 (P3) — `d2p doctor` 命令没在 dev doc / surface 完整定义

**Gap 2 已部分提及**。但 dev doc 把"`d2p doctor`"当成已存在的命令，**没说**这个命令到底:
- 接什么参数？
- 输出什么？
- 错误码？
- exit code 约定？

**修哪边**: **修 dev doc + surface (P3)**。`d2p doctor` 应该当一个完整 deliverable 写进 §A.7 或新增 §A.8。

## 4. 整体判断

**Subagent 严守黑盒原则**：
- 没读 `11-mvp-0.5-spec.md`（spec 文件）✓
- 没读 `daemon/src/` 任何代码 ✓
- 假设的 mock 库（vitest, memfs, nock, fast-check 等）属于通用工具不是 d2p 实现细节 ✓
- 提到的 filesystem path (`docs/dogfood/`, `.d2p/vision.md`) 都在 public surface 已暴露 ✓

**Test doc 质量**:
- 90 case 全在 Coverage Map 反查得到 Behavior ID
- 8 个错误码每个有 test
- §F PBT 覆盖 watchdog + sanity 的 invariant
- §H 诚实 ack 12 条
- 没踩 surface 边界（没假设 daemon framework / 库选择 / 内部数据结构）

**Truth gap 量化**:
- Surface 真缺 contract: 2 处（Gap 1 schema, Gap 2 doctor cmd trigger）
- Dev doc 描述 commit 缺 enforce 机制: 1 处（Gap 3 postmortem 30 天）
- UX quality 类 ack (合理): 4 处
- Governance 缺口 ack（已在 v0.7 §9.6 标识）: 2 处

**这是一次成功的"分离写 dev + test"实验**——subagent 用黑盒视角找出了 2 个真 surface gap（Gap 1 schema, Gap 2 doctor），如果 dev doc 写作者自己写 test 会很可能漏掉（因为知道实现里 schema 是什么样所以不会发现 surface 没暴露）。

这正是 Knight-Leveson 1986 N-version 独立性的文档层应用。

## 6. 这次实验的元结论

Knight-Leveson 1986 (N-version independent failure) 适用 doc 写作：**dev 写作者跟 test 写作者读不同 input 写出来的 doc，能挖出单一作者写不出来的真 contract gap**。

具体证据:
- Gap 1 (schema): dev 作者**知道**实现里 schema 是什么，不会觉得 surface 缺什么
- Gap 2 (doctor cmd): dev 作者**知道**这个 CLI 命令存在，不会觉得需要 surface 化
- 这两 gap 只有当**test 作者从外部用户视角推断"如果只看 public surface 能不能测"**时才浮现

**结论**: 这套"dev doc + 独立 subagent 写 test doc"工作流值得保留。建议 MVP-0 walking skeleton 之后所有 MVP 都跑这套：先写 dev doc，extract surface，subagent 写 test，比对找 gap。
