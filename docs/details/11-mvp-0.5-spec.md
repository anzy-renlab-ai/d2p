## §A. Cross-Engine Reviewer Default

### A.1 Motivation

参 `PRODUCT-HYPOTHESIS.md` v0.7 §3.1 + Round 2 R9：

- Littlewood-Miller 1989 ("Conceptual Modeling of Coincident Failures in Multiversion Software")：N-version diversity 收益**递减且强依赖 fault correlation**。
- MVP-0 default `claude --model X -p` 全 Claude family，Layer 2 / Layer 4 reviewer fault correlation 极高，**4 层 diversity 实质 ≈ 1 层**。
- MVP-0.5 强制 Layer 2 / Layer 4 **不同 LLM family** → 落地 v0.7 §3.1 + 锁定 §Q4.3 大厂 vendor-lock 不可达的护城河。

### A.2 Architecture

#### A.2.1 现有 reviewer pipeline (MVP-0)

```
artifact (worktree diff)
  → Layer 1: static gate (deterministic, no LLM)
  → Layer 2: alignment probe (claude haiku)
  → Layer 3: behavioral (preset runner, no LLM)
  → Layer 4: adversarial (claude sonnet/opus, 高敏 gap only)
  → done-check (claude haiku)
```

**问题**: Layer 2 + Layer 4 都用 Claude family，fault correlation ~80%。

#### A.2.2 MVP-0.5 改造

```
artifact
  → Layer 1: static gate (deterministic)
  → Layer 2: alignment probe (engine X)  // X ∈ {claude, codex, gemini}
  → Layer 3: behavioral (deterministic)
  → Layer 4: adversarial (engine Y, Y ≠ X)  // enforce different family
  → done-check (engine X)  // 可与 Layer 2 同 engine
```

**Hard constraint**: `layer2_engine_family !== layer4_engine_family`。

### A.3 配置 schema

新增到 `daemon/src/config/types.ts`:

```typescript
export type LLMEngineFamily = 'claude' | 'codex' | 'gemini'

export interface ReviewerEngineConfig {

  layer2_model: string                  // e.g. 'haiku-4-5' / 'gpt-5-mini' / 'gemini-3-flash'
  layer4_engine: LLMEngineFamily       // adversarial family
  layer4_model: string
  done_check_engine: LLMEngineFamily   // 可与 layer2 同
  done_check_model: string
}

// Default config（写在 default-config.ts）
export const DEFAULT_REVIEWER_ENGINES: ReviewerEngineConfig = {
  layer2_engine: 'codex',
  layer2_model: 'gpt-5-mini',
  layer4_engine: 'claude',
  layer4_model: 'opus-4-7',
  done_check_engine: 'codex',
```

**Config 验证**：load 配置时检查 `layer2_engine !== layer4_engine`，违反则 fail-fast 退出，错误码 `E_REVIEWER_SAME_FAMILY`。

### A.4 Subprocess wrapper

新建统一接口：

### A.6 First-run onboarding

启动 `d2p start` 时 (在 walking skeleton MVP-0 基础上扩展):

```
1. d2p 调 getEngine('claude').isAvailable() 
2. d2p 调 getEngine(default config.layer2_engine).isAvailable()
3. d2p 调 getEngine(default config.layer4_engine).isAvailable()

If layer2 + layer4 都不可用:
  → 显示 onboarding screen:
    "d2p 需要 2 家不同 LLM 提供商的 CLI。
     检测到: [cc 已装 / codex 未装 / gemini 未装]
     请装下列任一组合:
       (A) cc + codex
       (B) cc + gemini  
       (C) codex + gemini
     装好后回来重启 d2p。"
  → daemon refuse to start session
```

**Hard requirement**: 不允许只用一家 family 跑 d2p MVP-0.5。

### A.8 Cost 影响

| Scenario | MVP-0 (全 cc) | MVP-0.5 (cross-engine) | Delta |
|---|---|---|---|
| 单次 reviewer 跑（Layer 2+3+4） | $0.05-0.20 | $0.08-0.30 | +50% |
| 完整 1 个 gap loop（含 retry） | $0.30-1.50 | $0.45-2.25 | +50% |
| 完整 demo→product session | $2-15 | $3-22 | +50% |

### A.9 Failure modes

| Failure | Detect | Action |
|---|---|---|
| Engine 不可用 (CLI 没装) | `isAvailable()` returns false | First-run onboarding refuse to start |
| Engine API key 过期 | invoke 返回 stderr 含 "401" / "unauthorized" | UI 提示用户重新登录该 engine |
| Engine output 不符合 JSON schema | parse fail | 5 retry，仍失败 escalate |
| Engine timeout | duration > timeout_ms | retry 一次，仍 timeout 标 ESCALATE |
| Cross-engine 都挂了 | 两个 engine 都 fail | Loop hard stop，UI 显示 "engines unavailable" |

## §B. Spec Sanity Gate v1

### B.1 Motivation

参 v0.7 Q6.4 + Round 2 场景 1:

- Demo 本身有 bug → d2p 把 bug 当 spec 抄进 product
- 用户 vision 没写但其实在乎的 capability (e.g. Unicode 名字) → d2p 漏抓
- 双绿 = 错得整齐

### B.2 Design

Insert 在 vision elicit 之后、differ 之前：

```
vision elicit (with user)
  ↓
spec sanity gate (independent LLM, different frame)
  ↓ generate hostile use cases over 10 standard dimensions
  ↓ diff with vision text
  ↓ if mismatch > threshold → escalate to user
  ↓
differ (gap 生成)
```

### B.3 10 standard hostile-use-case dimensions

MVP-0.5 thin version：

| # | Dimension | 示例 hostile use case |
|---|---|---|
| 1 | Unicode handling | 用户名 `张三💩` / 文件名含 RTL / emoji in input |
| 2 | Authentication boundary | 未登录访问 / token expired / 跨用户访问 |
| 3 | Data validation | 空字符串 / 超长输入 / SQL injection / XSS / 控制字符 |
| 4 | Concurrency | 同一资源同时被多用户写 / race condition on counter |
| 5 | Timezone | UTC vs local / DST 切换 / 跨日期变更线 |
| 6 | Network | 慢网 / 断网 / 重试 / 重复请求 / partial response |
| 7 | Storage | 磁盘满 / 文件 corrupt / DB connection lost |
| 8 | Locale | 多语言 / 数字格式 (`1,000` vs `1.000`) / 货币 |
| 9 | Edge case input | 0 / 负数 / 上限 / 下限 / NaN / undefined / 极小 / 极大 |
| 10 | Privacy / data exfil | PII 不该泄露 / log 不该含 sensitive data / error message 不该暴露内部结构 |

### B.4 Implementation

新模块 `daemon/src/sanity-gate/`:

```
daemon/src/sanity-gate/
├── index.ts              # 主入口
├── dimensions.ts         # 10 dimensions 定义
├── prompt.ts             # LLM prompt template
├── parser.ts             # parse LLM JSON output
└── escalation.ts         # 决定是否 escalate user
```

主接口：

```typescript
// daemon/src/sanity-gate/index.ts
export async function runSpecSanityGate(
  vision: VisionDoc,
  demo: DemoSnapshot,
  config: SanityGateConfig
): Promise<SanityReport> { ... }

export interface SanityGateConfig {
  engine: LLMEngineFamily       // 必须 != vision elicit engine
  model: string
  uncovered_threshold: number   // default 5（10 dimension 里 ≥5 没覆盖就 escalate）
  skip_dimensions?: string[]    // 用户可显式跳过某些 dimension
}

export interface SanityReport {
  scanned_at: string  // ISO 8601
  covered: Dimension[]
  uncovered: Dimension[]
  escalation_needed: boolean
  user_escalation_message: string
  hostile_use_cases: HostileCase[]  // 每个 dimension 一个具体例子
}

export interface Dimension {
  id: string  // 'unicode' / 'auth' / 'validation' / ...
  name: string
  coverage_reasoning: string  // LLM 给的判定理由
}

export interface HostileCase {
  dimension_id: string
  scenario: string  // 自然语言描述
  expected_behavior: string  // LLM 推测的"如果 vision 在乎"应该怎么处理
  observed_in_vision: boolean
}
```

### B.5 Independence requirement

- Spec sanity gate engine **必须不同于** vision elicit engine。
- e.g. 若 elicit 用 cc，sanity gate 用 codex 或 gemini.
- 若 cross-engine 配置 (§A) Layer 2 engine = X，Layer 4 engine = Y，则 sanity gate engine = Layer 4 engine (Y)。这样跟 alignment probe 错开。

### B.6 Escalation flow

```
if SanityReport.escalation_needed:
  show blocking dialog:
    "We notice the vision doesn't address these dimensions: [list]
     For each, our independent reviewer generated a hostile use case:
     - {Dimension 1}: {scenario_1}
     - {Dimension 2}: {scenario_2}
     - ...
     
     For each dimension you can:
       (A) Add to vision (open editor)
       (B) Explicitly mark as out-of-scope (e.g. 'this demo doesn't handle Unicode names by design')
       (C) Continue without addressing"
  
  Wait for user input.
  Each choice gets logged into vision.md as appendix.
```

User 选 (C) 时**继续但记入 trajectory**——MVP-0.5 不强制必须 cover dimension，但留 audit trail。

### B.7 Failure modes

| Failure | Detect | Action |
|---|---|---|
| Sanity engine 不可用 | `isAvailable()` false | Skip sanity gate，warn 用户，记 trajectory |
| Sanity engine 超时 | duration > 120s | Retry 1 次，仍超时 skip |
| Sanity engine 输出 schema 错 | parse fail | 3 retry，失败 skip |
| 用户 dismiss 所有 escalation | uncovered > 0 + 用户全选 (C) | 继续但 vision.md 加 warning section |

### B.8 Cost

- 一次 sanity gate ~$0.05-0.20 (跑一次 LLM call 含 10 dimension 评估)
- 每个 d2p session 开头跑一次（不重复跑）
- 跟 cross-engine reviewer cost ($0.45-2.25 per gap × N gap) 比，sanity gate cost 是 noise

### B.9 What this does NOT do

显式 declare 不抓:
- Class C (concurrency / distributed protocol) bug 抓不到——dimension 4 只是 awareness 提示，不真验
- Class D 的子集（业务规则 / domain logic）——MVP-0.5 thin version 只覆盖 10 个 generic dimension，不覆盖 domain-specific
- vision 内部矛盾（e.g. "无登录" 但又 "保存用户数据"）——MVP-1+ 加

## §C. GitHub PR-Comment / Status-Check 集成

### C.1 Motivation

参 v0.7 §3.3 + FAANG Q1：dashboard-as-primary-UI 跟 PR-anchored 工作流冲突。把 reviewer 决策 surface 到 GitHub PR。

### C.2 Two integration points

1. **PR comment**: 每次 reviewer 决策（4 层任一层 PASS/FAIL）→ d2p 在对应 PR 上发 inline comment.
2. **Status check**: d2p 注册 4 个 status check (`d2p/static`, `d2p/alignment`, `d2p/behavioral`, `d2p/adversarial`)，PR merge gate 用。

### C.3 Implementation

新模块 `daemon/src/integrations/github/`:

```
daemon/src/integrations/github/
├── index.ts              # 主入口
├── client.ts             # GitHub API wrapper
├── auth.ts               # OAuth + token storage
├── pr-comment.ts         # 发 PR comment 逻辑
├── status-check.ts       # 注册 status check 逻辑
└── repo-detector.ts      # 检测 remote 是不是 GitHub
```

主接口：

```typescript

export interface GitHubIntegration {
  isEnabled(demoPath: string): Promise<boolean>
  postPRComment(repo: RepoRef, pr: number, comment: PRComment): Promise<void>

export interface PRComment {

  line?: number  // optional

  reasoning: string
}

  context: 'd2p/static' | 'd2p/alignment' | 'd2p/behavioral' | 'd2p/adversarial'
  state: 'success' | 'failure' | 'pending'

  target_url: string  // d2p dashboard link for details

```

### C.4 Auth model

- d2p 使用 **GitHub App token** (用户 OAuth 授权一次)
- App scope: `repo:status`, `pull_request:write`, `pull_request_review:write`
- App **不**要求 `contents:write` 到 main branch — 只能 push 到 feature branch + PR review
- Token 存在 `<demo>/.d2p/github-app.json`，**OS keychain 加密**:
  - Mac: keychain via `keytar`
  - Windows: Credential Manager via `keytar`
  - Linux: libsecret via `keytar`
- 若 `keytar` 不可用 (e.g. headless Linux server)：fail-fast，告诉用户安装 libsecret

### C.5 Repo detection

```typescript
// daemon/src/integrations/github/repo-detector.ts
export async function detectGitHubRepo(demoPath: string): Promise<RepoRef | null> {
  // 1. read .git/config
  // 2. parse [remote "origin"] url
  // 3. match against pattern: github.com[:/](.+)/(.+?)(\.git)?$
  // 4. return {owner, name} 或 null
}
```

若 detector 返回 null（remote 不是 GitHub）→ PR 集成 **silent skip**，d2p 继续跑，UI 只 dashboard。

### C.6 Branch + PR flow

```
1. d2p worktree spawn 在 .d2p-worktrees/<gap-slug>/
2. d2p git push 到 origin (or fork if no write perm)
3. d2p auto-open draft PR via GitHubIntegration.openDraftPR()
4. PR body 含:
   - gap description
   - vision quote (related portion)
   - d2p session ID + link to dashboard
5. 每层 reviewer 跑完，d2p:
   - postPRComment (若 FAIL, inline 到具体 file:line 若 reviewer 报；若 PASS, 简短 summary comment)
   - setStatusCheck (PASS → success, FAIL → failure, in-progress → pending)
6. 4 层全 PASS → status check 全绿 → PR 可 merge (d2p 自动 merge if §9 Q3.6 配置允许)
```

### C.7 Behavior 矩阵

| User repo state | d2p 行为 |
|---|---|
| GitHub remote + 用户授权 | 启用 PR 集成，draft PR + comments + status check |
| GitHub remote + 用户没授权 | UI 提示授权，授权前 fallback dashboard only |
| GitLab / Bitbucket remote | 不启用 PR 集成（MVP-0.5 only GitHub），dashboard only |
| 无 remote (local-only repo) | 不启用 PR 集成，dashboard only |
| 用户撤销 GitHub App | 下次跑时 detect 失败，fallback dashboard only + warn |

### C.8 Failure modes

| Failure | Detect | Action |
|---|---|---|
| GitHub API rate limit | response `403` + `X-RateLimit-Remaining: 0` | Exponential backoff (1m / 5m / 15m)，期间 fallback dashboard |
| GitHub App token revoked | response `401` | UI 强提示用户重新授权 |
| Network down | fetch throw | Local retry queue (in-memory)，network 恢复后 flush |
| Repo 是 private + d2p App 没装 | response `404` | UI 提示用户安装 App 到 repo |
| Status check 太多（rate limit per minute） | per-sha limit | Batch commit (一次提交 4 个 layer status) |

### C.9 Privacy guarantees

- d2p **不 leak source code 到 GitHub 之外**
- token 只在本地 keychain
- 用户 codebase 不上传任何第三方 server
- d2p dashboard URL 是 `localhost:5173/sessions/<id>`，**外部无法访问**
- PR comment 里的 reviewer reasoning 是 LLM 输出的判定理由，含部分 code reference（这是 PR review 的本来用途）

### C.10 Cost

- GitHub API 调用免费（OSS repo）
- 私有 repo 走用户自己的 GitHub plan
- d2p 自身**不收任何 GitHub-related 费用**

## §D. Dogfood-Debt Mode

### D.1 Motivation

参 v0.7 Q7.4 + Round 2 R6 + 场景 1: Founder dogfood 必须工程化 enforce，不能是文档承诺。

### D.2 Four Artifacts

#### D.2.1 `tools/dogfood-stall-watchdog.mjs`

Node.js script, ~150 LOC. 实现：

```javascript
import fs from 'node:fs'
import path from 'node:path'
import { parse as parseYaml } from 'yaml'

const DOGFOOD_DIR = path.resolve('docs/dogfood')
const STALL_THRESHOLD_DAYS = 5
const P1_ALERT_STALL_COUNT = 2

function parseFrontmatter(filePath) {
  const content = fs.readFileSync(filePath, 'utf-8')
  const match = content.match(/^---\n([\s\S]*?)\n---/)
  if (!match) return null
  return parseYaml(match[1])
}

function scanDogfoodTrajectories() {
  const files = fs.readdirSync(DOGFOOD_DIR)
    .filter(f => f.endsWith('.md') && !f.startsWith('_'))
  
  const now = new Date()
  const trajectories = []
  
  for (const file of files) {
    const filePath = path.join(DOGFOOD_DIR, file)
    const fm = parseFrontmatter(filePath)
    if (!fm) continue
    if (fm.status === 'complete' || fm.status === 'abandoned') continue
    
    const lastUpdate = new Date(fm.last_updated)
    const daysSince = (now - lastUpdate) / 86400e3
    
    let status = 'active'
    let stallRootCauseHypothesis = null
    if (daysSince > STALL_THRESHOLD_DAYS) {
      status = 'stalled'
      stallRootCauseHypothesis = inferStallCause(filePath)
    }
    
    trajectories.push({
      slug: fm.slug,
      file,
      status,
      days_since_update: daysSince,
      stall_root_cause_hypothesis: stallRootCauseHypothesis
    })
  }
  
  const stalledCount = trajectories.filter(t => t.status === 'stalled').length
  const p1Alert = stalledCount >= P1_ALERT_STALL_COUNT
  
  return {
    scanned_at: now.toISOString(),
    trajectories,
    p1_alert: p1Alert
  }
}

function inferStallCause(filePath) {
  // 简单实现: grep file 最后一段 ## heading + 提取关键词
  // MVP-0.5 thin version: 找含 "stuck" / "blocked" / "investigating" 的 line
  const content = fs.readFileSync(filePath, 'utf-8')
  const lines = content.split('\n').reverse()
  for (const line of lines) {
    if (/stuck|blocked|investigating|TODO/i.test(line)) {
      return line.trim().slice(0, 200)
    }
  }
  return null
}

// 主入口
const state = scanDogfoodTrajectories()
fs.writeFileSync(
  path.join(DOGFOOD_DIR, '_watchdog-state.json'),
  JSON.stringify(state, null, 2)
)

if (state.p1_alert) {
  console.error('P1 ALERT: ≥2 dogfood trajectories stalled')
  process.exit(1)  // CI fail
} else {
  console.log('OK: no P1 alert')
  process.exit(0)
}
```

#### D.2.3 `docs/dogfood/_INVARIANT.md`

写死规则文档（D.2.2 workflow 引用）：

```markdown
# Dogfood Invariant

## Rule
任何 trajectory 文件:
- 不以 `_` 开头
- frontmatter `status` 不是 `complete` / `abandoned`
- 最后 commit 更新 > 5 天前

**项目进入 dogfood-debt mode**。

## dogfood-debt mode 行为

### Prohibited
- 任何 `type/feat` label 的 PR 都无法 merge
  - 由 GitHub branch protection enforce: required check `dogfood-watchdog` 必须 success

### Allowed
- `type/dogfood-fix` label 的 PR — 修 dogfood 阻塞根因
- `type/docs` label 的 PR — 文档调整
- `type/chore` label 的 PR — 维护类

### Required
- 维护者必须先在 `docs/dogfood/_postmortems/<YYYY-MM-DD-slug>.md` 写复盘:
  - 哪个 trajectory stall
  - 为啥 stall
  - 根因
  - 下一步
  - 是否要改 d2p 设计

## Exit 条件

同时满足:
- stall trajectory 数 ≤ 1
- ≥ 1 个 trajectory 在最近 7 天有 commit
```

### D.3 GitHub branch protection 配置

仓库 settings → Branches → Branch protection rules for `main`:

```
Required status checks before merging:
  ✓ dogfood-watchdog
  ✓ (其他 CI 比如 npm test)

Require branches to be up to date: yes
Require pull request reviews: 1 (作者审 community PR)
```

dogfood-watchdog GitHub Action 把 `p1_alert: true` 时 fail，导致 status check FAIL，branch protection 阻止 merge。**这就是 enforce 落地——red line 是 GitHub 配置不是文字 axiom**。

### D.4 Onboarding

新维护者加入项目时：
1. 必读 `docs/dogfood/_INVARIANT.md`
2. 必须知道 `docs/dogfood/<slug>.md` 文件 frontmatter schema
3. 不能绕过 branch protection（仓库 settings 上禁用 admin override）

### D.5 Failure modes

| Failure | Detect | Action |
|---|---|---|
| Watchdog 自身 bug | tests/dogfood-invariant.test.mjs 抓 | tests 必须 pass 才能 ship watchdog 改动 |
| GitHub Action 跑失败 | workflow run status | retry，2 次都失败 → 发 maintainer email (via Actions notification) |
| 没有 trajectory 文件 | scanDogfoodTrajectories 返回 empty | 不 alert（不强制 dogfood，只在已有 trajectory 时 enforce） |
| Maintainer 想紧急 bypass | 改 branch protection 手动 disable required check | 只允许 owner 一人有权限；改动记进 audit log；改完必须 30 天内 write postmortem |

### D.6 Cost

- GitHub Action 跑成本 ~免费 (1 个 job × 30 秒 × 每天 1 次 = << GitHub Free tier 上限)
- 维护成本：watchdog 自身代码 ~150 LOC + 8 test = ~1 maintainer-day to ship
- Ongoing：当 stall 触发，每 stall trajectory 1-3 maintainer-day 写 postmortem + 修根因
