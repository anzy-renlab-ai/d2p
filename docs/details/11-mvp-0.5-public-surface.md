## §A. Cross-Engine Reviewer

### A.1 Public interface

```typescript
type LLMEngineFamily = 'claude' | 'codex' | 'gemini'

interface LLMSubprocOptions {
  model: string
  prompt: string
  timeout_ms?: number  // default 60_000
  max_retries?: number  // default 3
  expect_json?: boolean  // default false
}

interface LLMSubprocResult {
  ok: boolean
  raw_stdout: string
  raw_stderr: string
  parsed_output?: unknown
  duration_ms: number
  estimated_cost_usd: number
  retries_used: number
}

interface LLMSubproc {
  family: LLMEngineFamily
  invoke(opts: LLMSubprocOptions): Promise<LLMSubprocResult>
  isAvailable(): Promise<boolean>
}

interface ReviewerEngineConfig {
  layer2_engine: LLMEngineFamily
  layer2_model: string
  layer4_engine: LLMEngineFamily
  layer4_model: string
  done_check_engine: LLMEngineFamily
  done_check_model: string
}
```

### A.2 Default config

```typescript
DEFAULT_REVIEWER_ENGINES = {
  layer2_engine: 'codex',
  layer2_model: 'gpt-5-mini',
  layer4_engine: 'claude',
  layer4_model: 'opus-4-7',
  done_check_engine: 'codex',
  done_check_model: 'gpt-5-mini'
}
```

### A.3 Behaviors (要被 test 覆盖)

**B-A-1**: Config 加载时验证 `layer2_engine !== layer4_engine`。违反 → 抛错 `E_REVIEWER_SAME_FAMILY`，daemon fail-fast 退出。

**B-A-2**: `LLMSubproc.isAvailable()` 检查 CLI 是否装好且能跑（无 API key 也算 not available）。

**B-A-3**: `LLMSubproc.invoke()` 在 `expect_json: true` 时，若 stdout 不能 parse JSON，自动 retry 至多 `max_retries` 次。返回 `result.ok = false` + `retries_used = max_retries` 当全失败。

**B-A-4**: `LLMSubproc.invoke()` 超过 `timeout_ms` 后取消子进程并返回 `ok: false`，重试一次后仍 timeout 标 `ok: false`。

**B-A-5**: Daemon 启动时调用 `isAvailable()` for layer2_engine + layer4_engine + done_check_engine。任一不可用 → 进入 onboarding mode（显示安装提示），refuse start session 直到双 engine 可用。

**B-A-6**: 同一段 prompt 在两个不同 engine (e.g. codex vs claude) 上跑，**两次输出的 metadata** (model, family) 必须不一样；**raw_stdout 内容**不要求一致（不是单元测试，是 cross-engine 本意）。

**B-A-7**: `estimated_cost_usd` 必须为非负数字。`duration_ms` 必须为非负数字。

### A.4 Calibration check

```typescript
interface CalibrationFixture {
  id: string
  code: string
  ground_truth: 'buggy' | 'clean'
  context_vision: string
}

interface CalibrationReport {
  drift_percent: number  // 0-100
  engine_a_metrics: { pass_when_buggy_rate: number, fail_when_clean_rate: number }
  engine_b_metrics: { pass_when_buggy_rate: number, fail_when_clean_rate: number }
  warning_threshold_exceeded: boolean  // true when drift > 5%
}

function runCalibrationCheck(
  engines: [LLMSubproc, LLMSubproc],
  fixtures: CalibrationFixture[]
): Promise<CalibrationReport>
```

**B-A-8**: `runCalibrationCheck` 在 fixtures 同时给到两个 engine 时，返回 `drift_percent`。如果两个 engine 完全一致 → drift = 0。

**B-A-9**: `warning_threshold_exceeded` 当 `drift_percent > 5` 时为 true。

### A.5 Failure modes

| Failure | 表现 | Expected outcome |
|---|---|---|
| Engine CLI 未装 | `isAvailable()` returns false | daemon refuse start, 进 onboarding |
| Engine API key 过期 | invoke 返回 stderr 含 401/unauthorized 字样 | result.ok = false, error_kind = 'auth_expired' |
| Engine output 非 JSON (when expect_json=true) | parse fail | retry 至 max_retries, 然后 ok=false |
| Engine timeout | duration_ms > timeout_ms | retry 1 次, 仍超时 ok=false |
| 两 engine 都挂 | 两个 invoke 都 ok=false | loop 停, UI 显示 "engines unavailable" |

## §B. Spec Sanity Gate

### B.1 Public interface

```typescript
interface SanityGateConfig {
  engine: LLMEngineFamily  // 必须不同于 vision elicit engine
  model: string
  uncovered_threshold: number  // default 5
  skip_dimensions?: string[]
}

interface Dimension {
  id: string  // e.g. 'unicode', 'auth', 'validation'
  name: string
  coverage_reasoning: string
}

interface HostileCase {
  dimension_id: string
  scenario: string
  expected_behavior: string
  observed_in_vision: boolean
}

interface SanityReport {
  scanned_at: string  // ISO 8601
  covered: Dimension[]
  uncovered: Dimension[]
  escalation_needed: boolean
  user_escalation_message: string
  hostile_use_cases: HostileCase[]
}

function runSpecSanityGate(
  vision: VisionDoc,
  demo: DemoSnapshot,
  config: SanityGateConfig
): Promise<SanityReport>
```

### B.2 10 standard dimensions

固定 dimension ID 列表：
- `unicode`
- `auth`
- `validation`
- `concurrency`
- `timezone`
- `network`
- `storage`
- `locale`
- `edge_input`
- `privacy`

### B.3 Behaviors

**B-B-1**: `runSpecSanityGate` 总是返回恰好 10 个 dimension（在 covered + uncovered 中），除非 `skip_dimensions` 跳过——跳过的 dimension 既不在 covered 也不在 uncovered。

**B-B-2**: `escalation_needed = uncovered.length >= uncovered_threshold` (default 5)。

**B-B-3**: 若 vision 显式 mention 某 dimension（e.g. "支持 Unicode 用户名"），该 dimension 必须在 covered，**不能**在 uncovered。

**B-B-4**: `hostile_use_cases` 数组长度 = 10（一个 dimension 一个 case，包括 covered 的）。每个 case 的 `dimension_id` 唯一。

**B-B-5**: 若 `config.engine` 等于 vision elicit engine（即同 family），`runSpecSanityGate` 抛 `E_SANITY_GATE_SAME_ENGINE`。

**B-B-6**: `user_escalation_message` 是 markdown 字符串，包含 uncovered dimension 列表和每个的 hostile case 描述。

**B-B-7**: Sanity gate engine 不可用时（`isAvailable()` false），`runSpecSanityGate` 返回 `SanityReport` 含 `escalation_needed: false` + `covered: []` + `uncovered: []` + `user_escalation_message` 含 warning "sanity gate skipped, engine unavailable"——**不抛错，不阻塞 session**。

### B.4 Escalation flow (UI 行为)

**B-B-8**: 当 `escalation_needed: true`，UI 必须显示 blocking dialog 提供三选项：
- (A) Add to vision (打开 editor)
- (B) Mark dimension out-of-scope
- (C) Continue without addressing

**B-B-9**: 用户选 (B) 或 (C) 时，d2p 把决定 append 到 `<demo>/.d2p/vision.md` 的 appendix section（标 `## Sanity Gate Decisions`），含 dimension_id + user_choice + timestamp。

**B-B-10**: 用户选 (C) 后 differ + reviewer 继续跑，**不再次触发** sanity gate（仅 session 开头跑一次）。

### B.5 Failure modes

| Failure | Expected outcome |
|---|---|
| Engine 不可用 | skip gate, warn user, session 继续 |
| Engine timeout (> 120s) | retry 1 次, 仍超时 skip + warn |
| Engine output schema 错 | retry 至 3 次, 失败 skip + warn |
| `config.engine` == vision elicit engine | 抛 `E_SANITY_GATE_SAME_ENGINE` |

## §C. GitHub PR-Comment / Status-Check 集成

### C.1 Public interface

```typescript
interface RepoRef {
  owner: string
  name: string
}

interface PRComment {
  file?: string
  line?: number
  body: string  // markdown
  reviewer_layer: 'static' | 'alignment' | 'behavioral' | 'adversarial'
  decision: 'PASS' | 'FAIL'
  reasoning: string
}

interface StatusCheck {
  context: 'd2p/static' | 'd2p/alignment' | 'd2p/behavioral' | 'd2p/adversarial'
  state: 'success' | 'failure' | 'pending'
  description: string
  target_url: string
}

interface GitHubIntegration {
  isEnabled(demoPath: string): Promise<boolean>
  postPRComment(repo: RepoRef, pr: number, comment: PRComment): Promise<void>
  setStatusCheck(repo: RepoRef, sha: string, check: StatusCheck): Promise<void>
  openDraftPR(
    repo: RepoRef,
    branch: string,
    base: string,
    title: string,
    body: string
  ): Promise<{ pr_number: number }>
}

function detectGitHubRepo(demoPath: string): Promise<RepoRef | null>
```

### C.2 Behaviors

**B-C-1**: `detectGitHubRepo(demoPath)` 解析 `<demoPath>/.git/config`，匹配 `[remote "origin"]` URL pattern `github.com[:/](.+)/(.+?)(\.git)?$`。匹配 → 返回 `{owner, name}`；不匹配（或没 .git）→ 返回 null。

**B-C-2**: `isEnabled(demoPath)` returns true 当且仅当：
- `detectGitHubRepo` 返回 non-null
- 用户已授权 GitHub App（token 存在 keychain）
- Network 可达 (GET github.com responds)

**B-C-3**: `postPRComment`:
- 若 `file` + `line` 都给，发 inline file comment 到 PR 的 review thread
- 否则发 issue-level comment 到 PR
- comment body 含 `reviewer_layer` + `decision` 作 header（e.g. `### Layer 2 (alignment) — FAIL`）

**B-C-4**: `setStatusCheck`:
- context 必须是 4 个固定值之一
- state ∈ {success, failure, pending}
- `target_url` 必须是合法 URL（passes URL parsing）

**B-C-5**: `openDraftPR`:
- PR 创建为 draft (not ready for review)
- title / body 用 markdown
- 返回 `pr_number`

**B-C-6**: 若 `isEnabled = false`（remote 不是 GitHub / 没授权 / network down），`postPRComment` + `setStatusCheck` + `openDraftPR` **不抛错，silent no-op**（log warning 到 daemon log）。

**B-C-7**: 当 4 层 reviewer 全部 PASS，对应的 4 个 status check 都是 `state: success`。

**B-C-8**: 当任一层 FAIL，对应的 status check 是 `state: failure`，其他层若在跑则 pending。

### C.3 Auth model

**B-C-9**: GitHub App token 存在 OS keychain（Mac: keychain, Windows: Credential Manager, Linux: libsecret）。

**B-C-10**: 若 keychain 不可用（e.g. headless Linux 没装 libsecret），授权流程 fail-fast，提示用户安装 libsecret——**绝不**把 token 存明文文件。

**B-C-11**: 用户撤销 GitHub App 后，下次 `isEnabled()` 调用 returns false。

### C.4 Failure modes

| Failure | Expected outcome |
|---|---|
| GitHub API rate limit (403 + X-RateLimit-Remaining: 0) | exponential backoff (1m / 5m / 15m), 期间 fallback no-op |
| Token revoked (401) | UI 强提示用户重新授权 |
| Network down | local retry queue, 网络恢复后 flush |
| Repo private + App 没装 | 404 → UI 提示安装 App |
| Status check 超 per-minute limit | batch 4 个 layer status 一次提交 |

### C.5 Privacy guarantees (testable)

**B-C-12**: d2p **不上传 source code** 到 GitHub 之外的第三方 server。仅与 `github.com` API endpoint 通信。

**B-C-13**: PR comment 里可含 code snippet（标准 PR review 用途），但**不含**完整 source file。

**B-C-14**: dashboard URL（在 `target_url` 里）只是 `localhost:5173/sessions/<id>`，外部不可访问。

## §D. Dogfood-Debt Mode

### D.1 Public interface (watchdog)

```typescript
interface TrajectoryStatus {
  slug: string
  file: string
  status: 'active' | 'stalled'
  days_since_update: number
  stall_root_cause_hypothesis: string | null
}

interface WatchdogState {
  scanned_at: string  // ISO 8601
  trajectories: TrajectoryStatus[]
  p1_alert: boolean
}

function scanDogfoodTrajectories(): WatchdogState
```

### D.2 Frontmatter schema (trajectory files)

`docs/dogfood/<slug>.md` 文件必须含 YAML frontmatter:

```yaml
---
slug: <unique-slug>
status: in_progress | complete | abandoned
last_updated: <ISO 8601>
---
```

### D.3 Constants

- `STALL_THRESHOLD_DAYS = 5`
- `P1_ALERT_STALL_COUNT = 2`

### D.4 Behaviors

**B-D-1**: `scanDogfoodTrajectories()` 扫 `docs/dogfood/` 目录所有 `*.md` 文件，**忽略**以 `_` 开头的文件（如 `_INVARIANT.md`, `_watchdog-state.json`）。

**B-D-2**: trajectory file frontmatter `status: complete` 或 `status: abandoned` → 不计入 `trajectories` 数组。

**B-D-3**: `days_since_update = (now - last_updated) / 86400e3`. 计算结果为非负数。

**B-D-4**: `status: 'stalled'` 当 `days_since_update > STALL_THRESHOLD_DAYS`，否则 `'active'`。

**B-D-5**: `p1_alert: true` 当 stalled trajectories 数量 `>= P1_ALERT_STALL_COUNT` (= 2)。

**B-D-6**: `p1_alert: false` 当 stalled = 1 或 0。

**B-D-7**: 若 `docs/dogfood/` 目录不存在或没 `*.md` 文件，`trajectories: []`, `p1_alert: false`。

**B-D-8**: `stall_root_cause_hypothesis` 通过解析 file content 推断（regex 找 `stuck|blocked|investigating|TODO` 关键词的 line）。找不到返回 null。

**B-D-9**: Frontmatter 解析失败（无 `---` block 或 YAML 错）→ 跳过该文件，不计入 trajectories（不抛错）。

### D.5 _INVARIANT.md 规则

**dogfood-debt mode** 触发条件：watchdog 返回 `p1_alert: true`。

**dogfood-debt mode 下**:
- ❌ `type/feat` label 的 PR 不能 merge（required check `dogfood-watchdog` FAIL）
- ✅ `type/dogfood-fix` 或 `type/docs` 或 `type/chore` label 的 PR 可 merge
- 维护者必须在 `docs/dogfood/_postmortems/<YYYY-MM-DD-slug>.md` 写复盘

**Exit 条件** (同时满足):
- stall trajectory 数 ≤ 1
- ≥ 1 个 trajectory 在最近 7 天有 commit

### D.6 GitHub Action workflow

**B-D-10**: workflow 跑在 `cron: '0 3 * * *'` (UTC 03:00 daily) 和 `workflow_dispatch`。

**B-D-11**: workflow 输出 watchdog state 到 `docs/dogfood/_watchdog-state.json`。

**B-D-12**: 当 `p1_alert: true`：
- workflow exit code = 1（导致 required check FAIL）
- 自动 open GitHub issue 含 stall list + 推荐 action，label `dogfood-stall` + `priority/P1`

### D.7 Failure modes

| Failure | Expected outcome |
|---|---|
| `_INVARIANT.md` 缺失 | watchdog 不依赖它，正常跑（规则被 GitHub Action enforce） |
| Trajectory frontmatter 损坏 | 跳过该文件，不计入 |
| watchdog 本身 bug | tests/dogfood-invariant.test.mjs 抓 |
| GitHub Action 失败 | retry 2 次, 仍失败发 maintainer email |
| 维护者紧急 disable required check | 允许，但 audit log 记录，30 天内必须 postmortem |

## §E. 全局行为

### E.1 Hard requirements (跨模块)

**B-E-1**: MVP-0.5 daemon 启动时必须确认 2 家 LLM provider CLI 可用，否则 refuse start。

**B-E-2**: 任何 LLM 调用都通过 `LLMSubproc` 接口；**直接调 Anthropic SDK / OpenAI SDK 是禁止行为**（不持 API key，参 PRODUCT-HYPOTHESIS v0.7 红线 4）。

**B-E-3**: 用户 codebase 不上传到 d2p 控制之外的任何 server。

### E.2 Cost cap

**B-E-4**: per-session hard cost cap `$8`（从 MVP-0 的 `$5` 调整）。

**B-E-5**: per-iter hard cap `max_iter = 10`。

**B-E-6**: 触发任一 cap → graceful degradation 模式：
- 返回 `partial-PASS` 报告
- 列已完成 vs 未完成 capability
- 让用户决定 ship-partial vs continue-with-budget-raise（用户提高 cap 后继续）

### E.3 错误码（test 时验证）

| Code | Meaning |
|---|---|
| `E_REVIEWER_SAME_FAMILY` | layer2_engine == layer4_engine, config 拒绝 |
| `E_ENGINE_UNAVAILABLE` | 必需 engine CLI 没装或不能跑 |
| `E_SANITY_GATE_SAME_ENGINE` | sanity gate engine == vision elicit engine |
| `E_GITHUB_AUTH_REVOKED` | GitHub App token 被撤销 |
| `E_GITHUB_RATE_LIMIT` | API rate limit 命中，等 backoff |
| `E_KEYCHAIN_UNAVAILABLE` | OS keychain 不可用，拒绝存 token |
| `E_COST_CAP_EXCEEDED` | per-session $8 cap 命中 |
| `E_ITER_CAP_EXCEEDED` | max_iter = 10 命中 |

## §F. Test 范围建议（非强制，仅参考）

写 test doc 时**至少覆盖**：

- 每个 `Behavior B-X-Y` 至少 1 个 happy path test
- 每个 `Failure mode` 至少 1 个 negative test
- 每个 `Error code` 至少 1 个触发场景
- Cross-module integration:
  - Cross-engine reviewer (A) + Spec sanity gate (B) 共用同一 `LLMSubproc` registry——测同一 engine instance 不能被两边重复 invoke 时争抢资源
  - GitHub integration (C) + Reviewer (A) ——reviewer 决策落地后是否准确 surface 到 PR

可选: property-based test for `scanDogfoodTrajectories` (D)，输入随机 fixture 验证 invariant (e.g. p1_alert 与 stalled count 关系)。
