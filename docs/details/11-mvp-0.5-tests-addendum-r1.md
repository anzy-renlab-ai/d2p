## §H 还没法纯黑盒验的契约（delta）

R1 §H 列了 12 条。本轮新增 surface 引出以下需要承认的盲点：

| 契约 | 为什么难黑盒验 | 推荐 fallback |
|---|---|---|
| BSD sysexits exit code 64 (B-A-21) 在 Windows 上的语义 (T-A-21-invalid-component-exit-64) | Windows 进程退出码是 32-bit 无符号 ULONG，不强制 BSD sysexits 约定；cmd.exe / PowerShell 读出来仍是数值 64，但宿主 shell 可能截到 8-bit (`%ERRORLEVEL%` 在某些场景) | 单测捕获 child.exitCode 直接断言 === 64（在 Node child_process 层一致）；platform smoke 在 Win11 PowerShell + cmd.exe 各跑一次手动确认 `$LASTEXITCODE` / `%ERRORLEVEL%` 都是 64 |
| `d2p doctor` 自带 daemon spin-up (failure table) 真行为 (T-A-int-doctor-no-daemon-fallback) | "自动 spin up daemon (若可)" 是表述歧义——是 fork 一个守护进程还是 in-process run？黑盒只看得到 exit + stdout，看不到 daemon process 是否真起来 | 单测验逻辑路径（doctor 退出有结果就行）；smoke test 用 `ps` / Win 的 `Get-Process` 在 doctor 跑完后查 daemon PID 是否新增 |
| `--timeout 1` cancel 真正 kill 子进程 (T-A-22-timeout-yellow-on-hit) | 与 R1 §H "子进程被 kill 后无 orphan" 同源——Node child_process kill 行为跨平台差异 | 沿用 R1 §H fallback (platform smoke) |
| reasoning 截断点是字节还是字符 (B-A-12 / T-A-12-reasoning-over-2000) | "≤ 2000 字符" 在 UTF-16 surrogate pair (e.g. emoji) 上语义不清；surface 只说 "2000 chars" | 单测用纯 ASCII 验阈值；加 1 个 emoji-heavy reasoning fixture 作 acceptance smoke，确认不会把 surrogate 截半 |
| Schema validator entry point 真存在 (整 §A.5-tests 前提) | 公开 surface 暴露 schema 形状 + 「schema 违规触发 retry」，但没暴露 validator function name | test 直接断言 LLMSubproc.invoke 结果 + error_kind，不调内部 validator——所有 §A.5-tests 都从 invoke 的 result shape 反推 |
| `overall_status` 三态推导规则 (T-A-17-no-arg-*) | surface 给了 exit code (0/1/2) 但没明示 "overall_status === 'red' 当且仅当任一 check red"——只能从 exit-code 映射反推 | 单测验 exit + overall_status 一致性；如果未来 spec 改 derive 规则，需重测 |

**结论**: Addendum 新加的 13 个 behavior 比 R1 §A.1-4 更接近 user-visible CLI 行为，黑盒可测面更大。盲区集中在 ① 跨平台进程语义 ② surface 没暴露的 derive 规则。fallback 与 R1 §H 第 3/4/8 条复用即可（platform smoke matrix + canary）。
