## Section 1 — 战略 pivot 决策

### 从 v0.7 到 hardener-only

v0.7 PHYP 写的是「AI 时代的测试程序员 + demo→product 全闭环」——既要做底座（cross-engine reviewer / preset / evidence bundle），又要做闭环应用（demo→product fix loop）。今天收紧到**只做 hardener CLI + 3 底座**：

- **Protocol-1 — Cross-Engine Reviewer Protocol**：JSON Schema + reference impl，第三方可以喂任何 LLM engine 出 verdict
- **Protocol-2 — Preset Framework**：markdown-based check 定义 + plugin loader，第三方可以发自己的 preset 包
- **Protocol-3 — Evidence Bundle Format**：审计轨迹 JSON Schema，对接 Vercel / Netlify Deployment Check 之类的部署门
- **应用层 — `zerou audit` hardener CLI**：扫 vibe-coded repo (Lovable / Bolt / v0 导出) 出 finding list，可选 `--apply` 修

### 渐进 pivot，不硬切

- 旧 demo→product 代码（`orchestrator/loop.ts` 901 LOC + `agents/{detector,differ,implementer,vision,done-check}.ts` + UI mock 8 文件）挂 advanced mode 不删
- 第一波公网卖点全部围绕 hardener CLI 一条线
- 砍代码分波次，第 4 周才动 ui/src/mock/*

### BYO-key 升 default

v0.7 红线 4 写「默认 CLI subprocess（用户已付 Claude/ChatGPT 月费）」，今天承认这条押错市场：

- 调研发现 vibe coder（Lovable / Bolt / v0 / Cursor 用户）大多数没装 `claude` / `codex` / `gemini` CLI
- 他们付的是 ChatGPT Plus / Claude.ai Pro **web 订阅**，不是 API
- HTTP+key 不是 fallback，是这个用户群的 **default surface**

代码层早就支持：`daemon/src/engines/{openai-compat,anthropic-api}.ts` 已经能跑 HTTP+key，user 全局 `~/.d2p/config.json` 实际就是 MiniMax HTTP。**v0.7 写 hypothesis 之前没扫现有代码**，是踩坑。

### 触发条件

今天早上 (5/25) 5 个 subagent 跨判 + 3 路 fact-grounded 调研，发现 cross-engine reviewer 在 2026 已经 commodity：

- Superpowers (124K stars) 把 multi-model review 做成 OSS workflow
- Qodo 2.0 整套 review pipeline 直接卖
- GitHub Copilot Code Review 2026/6 起算 Actions minutes，等于内置
- 同时 88% autonomous agent pilots fail（Gartner 2026Q1）
- Lovable 单家就被发 BOLA 170 个 app（Escape.tech 报告）
- Escape.tech 扫 5600 个 vibe-coded app / 出 2000 个 critical vuln
- Fiverr 已经存在「vibe coding cleanup specialist」职业（$60-200/h）

→ 「**vibe coder 不会自己审 + 审完不会修**」是真痛点，cross-engine reviewer 已经不是 wedge 而是底座；wedge 是 hardener。

### 决策方法学回顾

今天这次 pivot 决策走的不是「一个人拍脑袋」，而是先并发派 5 个独立 subagent 用同一份问题集做跨判（互相不可见对方输出），出 5 份独立 5-12 页判断书；然后人工抽取共识点（5 份里至少 3 份命中）作为锁定结论，分歧点全部列为「需要现实数据回答」入第二轮调研；第二轮 3 路 fact-grounded 调研分别盯 Superpowers / Qodo / GitHub Copilot 三家做横评、Lovable / Bolt / v0 三家做用户群盘点、Escape.tech / Fiverr 做痛点 evidence。这套方法 cost ~$0.8（haiku tier 跑），出来的 framing 远比单线 grill 稳定。教训：定位级别决策不靠 grill 一个人的对话，靠 N 路独立判断 + 现实数据交叉。

### 把决策固化为单一句话

如果让我用一句话写今天的 pivot 结论给三个月后的自己看：「**不要做闭环，做开放契约 + 一个标杆应用**」。闭环（demo→product）是「我帮你做完」，标杆应用（hardener CLI）是「我帮你审完」——前者是承诺一切都解决，后者是承诺一个具体痛点解决得最干净。AI 时代用户不缺「自动化全流程」承诺（卖这个的太多），缺的是「这个具体小问题，我比谁都做得好」的窄门。Hardener 是这扇窄门。3 底座是为了让窄门后面的空间不被自己锁死，留给第三方一起做。

### 决策没有覆盖的两个开放问题

1. **第一批付费用户是谁不清楚**：hardener CLI 是 OSS-first，但盈利路径（hosted version / pro preset bundle / Vercel Marketplace / GitHub App billing）4 选 1 没拍。今天的判断是「先把 wedge 做出来，分发跑通了再谈定价」，但这是 punt 不是 answer。Week 3 真实 Lovable dogfood 跑完再回来回答。
2. **协议 3 (Evidence Bundle) 跟 Vercel / Netlify 的真实对接路径不明**：Vercel Deployment Check API 是 closed 协议，第三方接入要走他们的 Integration Marketplace 审核；Netlify Build Plugin 是 open 但 install base 小。今天判断是 Week 4 做 PoC 后再定 v2 路线，不在第一阶段押。

## Section 5 — Q4 7 条怀疑

**我看完代码后认同 pivot 方向, 但 framing 暗示的"复用 + 增量"不成立 —— 实际是 30% 复用 + 70% 重写或砍**。

具体证据:

1. **UI 13K LOC 几乎全废**: `ui/src/mock/*` 2K LOC 是 demo→product dashboard mock, `pages/{Setup,Workspace,Done}.tsx` 全是 demo→product 流程。Hardener UI 是 finding-list-centric, shape 完全不同。**预算: 6-8 week UI 重写**。pivot framing 没说这是 UI 重写。

2. **daemon 主循环 (`orchestrator/loop.ts` 770 LOC) 不能复用**: demo→product loop 是 "vision → differ → implementer → reviewer → done-check → merge"。Hardener 是 "scan → finding → cross-engine review → apply"。**主循环重写**。

3. **3 底座 + 1 应用 6 月做不完**: 单 maintainer 估 协议 2 (1 mo) + 协议 1 (1 mo) + hardener CLI alpha (2 mo) + dogfood/iter (2 mo) = 6 mo 边缘可行 **且 0 buffer**。协议 3 (Evidence Bundle Vercel/Netlify adapter) 推到 v2 (7-9 mo+)。

4. **顺序建议反过来**: Protocol-2 (Preset Framework) 已经在 v0 (loader + 6 preset 跑过), 最 mature。Protocol-1 (router.ts 60 行) 最容易抽。Protocol-3 (Evidence Bundle) 最复杂 + 无 prior art (Vercel/Netlify Deployment Check 是 closed API)。**建议: 2 → 1 → hardener → 3** (3 由 hardener dogfood 反推 schema, spec-first 听好但工程上虚)。

5. **"协议 1 commodity" 反驳成立性中等**: Superpowers/Qodo 是 application-layer, **没人发 JSON Schema 给第三方对接**。d2p 抽 router 出 protocol spec 仍有 first-mover 窗口, 但 ≤6 月。Linear Agent API 协议已经在做类似事, **真威胁**。

6. **Hardener vs demo→product 用户群只在 hackathon/indie 重叠, 整体分裂**: demo→product 用户在 "想 ship" 阶段; hardener 用户在 "ship 后怕翻车" 阶段。同一个 indie dev 不同时刻是不同 persona。**pivot 后 demo→product 不能"advanced mode"装饰, 实际是 fork** —— 这部分 sunk cost 要明示接受。

7. **BYO-key 升红线 4 是承认押错**: 大多数 vibe coder 没装 cc/codex CLI, 他们付 ChatGPT/Claude.ai 月费 + 用 Lovable/Bolt。**HTTP+key 是 vibe coder default 不是 fallback**。改对了, 但 v0.7 当时定 "默认 CLI subprocess" 是反市场。

8. **Evidence Bundle GDPR/SOC2 framing 在小项目上虚**: 合规对接是 customer-by-customer 律师 review, 不是 schema 定义。开放 spec 对 enterprise 没价值 (他们要 vendor 兜底 SLA + indemnification)。**§9.1 选 A "开源不变现"跟"满足 GDPR/SOC2"语义冲突** —— 付费方不存在。建议: Evidence Bundle framing 砍合规、留 "可重放审计" 一条就够。

**最该警惕的一件事**: walking skeleton smoke 昨天才 PASS, cross-engine enforce PR #3 今天才合。这两块 ~1 month 实际工程进展进了"advanced mode 冰柜"。**pivot 不是技术问题, 是承认 1 月+ 工程投入沉没成本**。如果接受, pivot 该上; 如果心痛, 先做 Wave 2 (hardener-relevant Spec sanity gate 改名 + 复用) 边修边 pivot 减少阵痛。

### 元教训：踩坑的共同 pattern

回看以上 10 条，有 4 个 pattern 反复出现：
1. **「假设环境跟我一样」是默认错觉** — 坑 #1 (smoke 读 user config)、坑 #6 (没扫现有代码就写 hypothesis)、坑 #5 (假设 bash cwd 跟主进程一致) 都是这个 pattern。修法：写前先 `git ls-files` + 跑现有 config 看实际默认值，不要凭印象。
2. **「test 看起来在测」≠「test 真的在测」** — 坑 #2 (auth-hint assert 字符串 stderr 已存在)、坑 #10 (loop cap 写了但默认值过宽等于没限) 同 pattern。修法：black-box test 要先确认目标信号不在 baseline 出现；任何 cap 类配置写完跑一次 trigger case 看真的会停。
3. **「文档说有」≠「surface 上能找到」** — 坑 #3 (engine JSON schema 没暴露)、坑 #4 (`d2p doctor` CLI 没暴露) 同 pattern。修法：CLI 命令 / HTTP endpoint / config 字段是 contract，文档写了就要 e2e 测试覆盖一遍。
4. **「Windows 跟 Linux 行为差异」无法假设** — 坑 #5 (worktree 嵌套 + file lock)、PR #2 静默问题（Windows 上 codex CLI 默认 PATH 不含 npm global bin）。修法：所有 path / process / git 操作都假设 Windows 会以你预料外的方式失败，写 fallback 路径。

## Section 10 — orchestrator/loop.ts 901 行核心流程文字版

读这一节，你不必打开 `daemon/src/orchestrator/loop.ts`。

### 入口

`runLoop(deps: LoopDeps, sessionId: number): Promise<void>` 是唯一入口。`deps` 含 `queries / db / inferredChecks?`。流程：取 session + demo → `ensureRepo(demoPath)`（确保是 git 仓）→ status 转 SETUP→LOOPING（或 PAUSED→LOOPING 发 `LOOP_RESUMED`）→ emit `LOOP_STARTED` + `startWatching` → 加载 vision.md → 读 loop caps（wallClock / maxConsecutiveEscalates / hardCostUsd / maxIterations）→ 进 for 循环。

### 主循环每一轮

1. **pause check** — `loopController.pauseRequested()` → 转 PAUSED + emit `LOOP_PAUSED reason:user_request` + 返
2. **wall-clock cap** — `Date.now() - loopStartTs > wallClockMs` → 转 PAUSED + emit `LOOP_PAUSED reason:wall_clock_cap` + 返
3. **hard cost cap** — `q.costTotals(sessionId).estimatedUsd >= hardCostUsd` → emit `LOOP_PAUSED reason:cost_cap` + 返
4. **escalate streak cap** — `escalateStreak >= maxConsecutiveEscalates` → emit `LOOP_PAUSED reason:consecutive_escalates_cap`
5. **runDifferAndPersist(ctx)** — 调 `agents/differ.ts` 出 gap list，写 SQLite，emit `DIFF_PRODUCED`；连续 K 次出 0 个新 gap 走 done-check
6. **取 next gap** → `processGap(ctx, gap)` 处理单个 gap
7. **done-check**（gap 为空 + emptyDifferStreak ≥ 阈值）— `agents/done-check.ts` 出 vision verdict；YES → emit `SESSION_DONE` + 返；NO → 继续循环

### processGap 单 gap 处理

1. emit `GAP_PICKED` → for `attempt < K`（K 由难度决定）：
2. `setupWorktree()` — 在仓外 `.worktrees/<slug>-<attempt>` 创 worktree，emit `WORKTREE_CREATED`
3. `agents/implementer.ts` 跑出 diff + commit；失败 K 次 → `GAP_ESCALATED reason:IMPLEMENTER_FAILURES`
4. emit `FIX_COMMITTED` → `static-gate/check.ts` 跑（typecheck/build/test）；fail → `STATIC_GATE_FAILED`，连续失败 → `GAP_ESCALATED reason:STATIC_GATE`
5. **alignment probe** — `agents/reviewers.ts` 调 critic engine（haiku probe，经 `engines/registry.ts getCriticEngine()`）打分；score 太低 → `GAP_ESCALATED reason:ALIGNMENT_LOW`
6. **behavioral review** — sonnet/opus tier，verdict APPROVE / NEEDS_FIX / BUGGY / SCOPE_CREEP / ARCHITECTURAL；非 APPROVE → 进 cross-engine second opinion（critic engine family 必须 ≠ worker family，由 `router.ts pickCriticEngine` 保证）
7. **cross-engine second opinion** — second critic 同意打回 → escalate；不同意 → 重试 implementer
8. **adversarial reviewer**（高敏 gap，severity ≥ 阈值才上）— `agents/reviewers.ts adversarial`；找到 break → `GAP_ESCALATED reason:ADVERSARIAL_BREAK`
9. **merge** — `git/merge.ts` 把 worktree commit merge 进 main → emit `MERGED` + `GAP_DONE`
10. **push & PR**（如启用）— `pushAndOpenPR()` 先正常 push；rejected → cherry-pick 到 `d2p/auto-fix/<slug>-<ts>` 分支 → `github/client.ts` 开 PR；冲突 → `GAP_ESCALATED reason:MERGE_CONFLICT`

### 辅助 function

- `loadVisionMd(session, demoPath)` — 读 `.d2p/vision.md`
- `writeVisionMd(demoPath, body)` — 写回
- `finalizeVisionFile(session, ...)` — emit `VISION_FINALIZED path`
- `pickK(difficulty)` — 难度→重试次数表
- `presetStatusSummary(items)` / `historyText(rows)` — 给 differ prompt 用的上下文

### 状态机

`SETUP → LOOPING → DONE | PAUSED | NEED_HUMAN`；PAUSED 可由 user pause / wall-clock / cost cap / escalate streak / max iterations 触发。

### 主要 SSE 事件清单

`SESSION_STARTED / LOOP_STARTED / LOOP_RESUMED / LOOP_PAUSED / VISION_QUESTION_ASKED / VISION_FINALIZED / DIFF_PRODUCED / GAP_INSERTED / GAP_PICKED / GAP_STARTED / WORKTREE_CREATED / FIX_COMMITTED / STATIC_GATE_PASSED / STATIC_GATE_FAILED / ALIGNMENT_RESULT / REVIEW_VERDICT / ADVERSARIAL_RESULT / GAP_ESCALATED / FIX_DROPPED / MERGED / GAP_DONE / DONE_CHECK_RESULT / SESSION_DONE / ERROR`。

### 跟 5 个 engine 的关系

- `differ / implementer` 用 **worker engine**（cfg.engine）
- `alignment probe / behavioral / adversarial / done-check` 用 **critic engine**，经 `engines/registry.ts getCriticEngine()` → `router.ts pickCriticEngine(worker, critic)` 出 `CriticPolicy`，保证 critic family ≠ worker family（除非 user 显式同 family + `D2P_ALLOW_DEGRADED_REVIEWER=1`）
- 所有 engine 通过 `engines/factory.ts` build → 走 `subproc/spawn.ts` 调 CLI 或 fetch HTTP → `engines/json-parse.ts` 抗污染解析返 JSON

### Pivot 后这段流程怎么改

Hardener 不需要这套 vision-driven 闭环。新的 `zerou audit` 流程预计是：
1. CLI 入口（`cli/src/index.ts` 加 `audit` 子命令）→ 解析 `<path>` + `--preset` + `--apply` flag
2. 调 `preset/loader.ts` 拉对应 category 的 check 列表（preset auto-detect 复用 `agents/detector.ts` 一小段，剩下砍）
3. for each check：跑 `static-gate/check.ts` 或 LLM-assisted（走 worker engine `engines/factory.ts`）出 raw finding
4. raw finding 进 **per-finding cross-engine review**：critic engine 拿同样 finding + repo context，二次判定 confirmed / false-positive / needs-context
5. confirmed finding 进 report；`--apply` mode 调 implementer 改代码 + 出 diff（**不直接 commit，留给 user review**）
6. 全程不进 SQLite session 表（hardener 单次跑，无 resume），日志走 stdout + 可选 `--out report.json` 出 Evidence Bundle JSON

这条路径上 `orchestrator/loop.ts` / `agents/{differ, vision, done-check}.ts` / `recovery/*` 全部不参与。`agents/reviewers.ts` 改成单独的 `hardener/cross-engine-review.ts`（接口换：从 "verdict on diff" 改 "verdict on single finding + repo evidence"）。`engines/*` + `preset/*` + `static-gate/*` + `subproc/*` + `storage/*` 是直接复用的核心 6 个模块。
