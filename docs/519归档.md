## 产品化 — 把手动 PR 流程加进 d2p

> 真跑里我手动做了：fetch origin → cherry-pick fix commit 到新分支 → 手写 PR body 列 NEED_HUMAN → `gh pr create`。这套动作产品化进 daemon。

提交：`1b55f34 feat(github-pr): auto-PR flow productized — body template + cherry-pick fallback + UI chip`

三件事：

1. **PR body 模板** — `daemon/src/orchestrator/pr-body.ts` 新文件 + `pr-body.test.ts` 6 个快照测。结构：
   - block 1：gap slug + severity + category + body
   - block 2：reviewer 评分（alignment score + behavioral APPROVE）
   - block 3：**NEED_HUMAN 列表**——同 session 被 reviewer 打回的 gap，每行 `slug · reasonCode · title`，reason code 表覆盖 13 种（OK / BUGGY / INCOMPLETE / SCOPE_CREEP / STATIC_GATE / IMPLEMENTER_FAILURES / K_EXHAUSTED / NON_JSON / ALIGNMENT_LOW / ADVERSARIAL_BREAK / CONFLICT / PR_PUSH_FAILED）
   - block 4：cost footer + `.d2p/session-summary.md` ref
   - 新 queries：`latestAlignmentScoreForFix(fixId)` / `latestReasonCodeForGap(gapId)`

2. **冲突自动 cherry-pick** — `daemon/src/git/push.ts` 新 `pushFixBranchOrCherryPick()`：先正常 push，stderr 含 `rejected` + `non-fast-forward` 时降级：
   - `git fetch origin <baseBranch>`
   - 起新分支 `d2p/auto-fix/<slug>-<unix-ts>` from `FETCH_HEAD`
   - `git switch` 过去 → 依次 `cherry-pick` 每个 work commit
   - 完了 `git switch` 回原 branch（不打扰用户 checkout）
   - cherry-pick 冲突时 `--abort` + 删半成品分支 + 返 `CONFLICT` 给 loop，loop 把 gap 升 NEED_HUMAN（reasonCode `CONFLICT`）
   - 3 个 vitest 用本地 bare remote 验：ff path / non-ff fallback / conflict path

3. **UI Commits 时间线 PR 链接 chip** — `daemon/src/types.ts` 的 `MergedCommitRow` 加 `prNumber` + `prUrl`，`listMergedCommits` SELECT `f.pr_number / f.pr_url`（fixes 表早就有，没暴露）。`ui/src/components/CommitsTimeline.tsx` 加 `🔗 PR #N` chip，coral 色，hover 反色，跳 GitHub。

## 后续 / 留待项

### 真跑暴露的更深层问题

- **alignment / behavioral reviewer 严格度**：CSRF 保护 / 密码恢复 / 备份这类没有 acceptance 可跑的 gap，behavioral reviewer 几乎全打回。要么放宽 acceptance（让 reviewer 接受 "code path 存在 + unit test 覆盖" 作为通过），要么差异化 preset item 的 reviewer 严格度
- **minimax 通过率低**：alignment probe 用的 minimax 模型对中文 gap 描述的对齐打分偏严，sonnet baseline 对比试一下，或者切 haiku 多次平均

### d2p loop 死循环（没全绿就一直找新 gap）

试验场跑到 1h31min 停下来是用户手停的，不是 loop 自终止。`vision satisfied: false` 时 loop 应当：
- 阶段性结束 session（连续 K 个 gap 都 NEED_HUMAN 就 `SESSION_PAUSED`）
- 或限 cost cap（$5 / $10 用户配）
- 或限 wall-clock（2h / 4h）
当前三个都没有，"无限刷新 gap" 是 P1 bug，但 19 号没修。

### session-end 汇总 PR vs 每 fix 一 PR

- 当前 **per-fix mode** 已支持（`github-pr` 模式每个 fix 一个 PR）
- 未来要加 **session-end mode**：session 结束打一个总 PR，把所有 merged fix 装进去
- 改 `pushAndOpenPR` 加 `mode: 'per-fix' | 'session-end'` 选项，UI 让用户在 session 配置时选

### 早上 plan 的 Out of scope 也没动

- diff 高亮没引 react-syntax-highlighter（纯 CSS tokenize 也没真做，目前就是 monospace 上色行）
- core-paths AI 推断（`.d2p/core-paths.inferred.json` 缓存）还是 mock，没真调 haiku
- SessionResumeBanner 在 demo 模式 demo 一次，真模式连不上 `session_resume_marks` 表（migration 落地了，写入路径还没接 loop pause 事件）

### 早上 plan 的 Out of scope 写明的
